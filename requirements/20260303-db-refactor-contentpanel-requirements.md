## 1. 现状分析

### 1.1 数据库架构现状

**当前实现** (`src/lib/db/connection.ts`):
- better-sqlite3 直接在 Renderer Process (Next.js) 中使用
- 数据库连接通过 `getDb()` 单例模式管理
- 数据库文件位置: `~/.lumos/lumos.db`
- 使用 WAL 模式提升并发性能

**存在的问题**:
1. **版本冲突风险**: better-sqlite3 是原生模块，需要针对 Node.js 和 Electron 分别编译
   - 开发环境 (Next.js dev server): 使用 Node.js ABI
   - 生产环境 (Electron renderer): 需要 Electron ABI
   - 当前通过 `scripts/after-pack.js` 在打包时重编译，但开发体验不佳

2. **安全性问题**: Renderer Process 直接访问文件系统和数据库
   - 违反 Electron 安全最佳实践
   - 如果引入第三方脚本，可能导致数据泄露

3. **架构不清晰**: 数据库逻辑与 UI 逻辑耦合
   - API 路由 (`src/app/api/*`) 直接调用 `src/lib/db/*`
   - 难以实现数据访问权限控制
   - 测试困难（需要模拟整个数据库环境）

### 1.2 ContentPanel (RightPanel) 现状

**当前实现** (`src/components/layout/RightPanel.tsx`):
- 固定显示文件树 (FileTree 组件)
- 宽度固定为 288px
- 支持折叠/展开
- 功能单一，无法扩展

**存在的问题**:
1. **功能局限**: 只能显示文件树，无法复用于其他内容
   - 飞书文档预览需要单独实现 (`FeishuPanel.tsx`)
   - 设置页面在独立路由 (`/settings`)
   - 未来的知识库、插件管理等功能无处安放

2. **用户体验不佳**: 缺乏统一的内容展示区域
   - 用户需要在不同页面间跳转
   - 无法同时查看多个内容（如对话 + 文档）
   - 缺少标签页管理

3. **代码重复**: 每个内容类型都需要实现自己的面板组件
   - `RightPanel.tsx` - 文件树
   - `FeishuPanel.tsx` - 飞书文档
   - 未来还需要 `KnowledgePanel.tsx`、`PluginPanel.tsx` 等

## 2. 问题定义

### 2.1 为什么要重构数据库架构？

**核心目标**: 将数据库操作迁移到 Electron Main Process，实现进程隔离

**解决的问题**:
1. **消除版本冲突**: Main Process 只需要 Electron ABI，Renderer Process 不再依赖原生模块
2. **提升安全性**: 遵循 Electron 安全最佳实践，Renderer Process 通过 IPC 访问数据
3. **改善架构**: 清晰的分层架构，Main Process 作为数据层，Renderer Process 作为表现层
4. **便于测试**: 可以独立测试数据库逻辑和 UI 逻辑

### 2.2 为什么要实现ContentPanel统一容器？

**核心目标**: 将工作区第3列改造成支持多标签的统一内容展示容器

**解决的问题**:
1. **功能扩展性**: 统一的容器可以承载任意类型的内容（文件树、文档、设置、知识库等）
2. **用户体验**: 标签页管理，用户可以同时打开多个内容并快速切换
3. **代码复用**: 避免为每个内容类型实现独立的面板组件
4. **未来规划**: 为知识库、插件管理等新功能预留扩展空间

## 3. 解决方案概述

### 3.1 数据库架构重构方案

**方案**: 将数据库操作迁移到 Electron Main Process，通过 IPC 通信

**架构设计**:
```
┌─────────────────────────────────────────┐
│         Renderer Process (Next.js)      │
│  ┌───────────────────────────────────┐  │
│  │  API Routes (/api/*)              │  │
│  │  - 调用 IPC 客户端                 │  │
│  └───────────────────────────────────┘  │
│                  ↓ IPC                  │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│         Main Process (Electron)         │
│  ┌───────────────────────────────────┐  │
│  │  IPC Handlers                     │  │
│  │  - 接收 IPC 请求                   │  │
│  │  - 调用数据库服务                  │  │
│  └───────────────────────────────────┘  │
│                  ↓                      │
│  ┌───────────────────────────────────┐  │
│  │  Database Service                 │  │
│  │  - better-sqlite3                 │  │
│  │  - 数据库操作逻辑                  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**关键技术点**:
1. **IPC 通信**: 使用 `ipcMain.handle` 和 `ipcRenderer.invoke`
2. **类型安全**: 定义统一的 IPC 接口类型
3. **错误处理**: IPC 调用失败时的降级策略
4. **性能优化**: 批量操作、缓存策略

### 3.2 ContentPanel统一容器方案

**方案**: 混合渲染 - 标签页管理 + 动态内容渲染

**架构设计**:
```
┌─────────────────────────────────────────┐
│         ContentPanel (统一容器)          │
│  ┌───────────────────────────────────┐  │
│  │  TabBar (标签栏)                   │  │
│  │  - 标签列表                        │  │
│  │  - 新建/关闭标签                   │  │
│  │  - 标签切换                        │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │  ContentRenderer (内容渲染器)      │  │
│  │  - 根据标签类型渲染对应组件         │  │
│  │  - FileTree / FeishuDoc / Settings │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**支持的内容类型**:
1. **文件树** (FileTree) - 当前已有
2. **飞书文档** (FeishuDoc) - 从 FeishuPanel 迁移
3. **设置** (Settings) - 从 /settings 路由迁移
4. **知识库** (Knowledge) - 未来功能
5. **插件管理** (Plugins) - 未来功能

**关键技术点**:
1. **状态管理**: 使用 Context 或 Zustand 管理标签状态
2. **动态渲染**: 根据标签类型动态加载对应组件
3. **性能优化**: 标签内容懒加载、虚拟滚动
4. **用户体验**: 标签拖拽排序、快捷键切换

## 4. 验收标准

### 4.1 数据库架构重构验收标准

**功能验收**:
- [ ] 所有数据库操作通过 IPC 调用，Renderer Process 不直接访问数据库
- [ ] 现有功能正常工作（会话管理、Provider 配置、MCP 配置、Skills 管理）
- [ ] 数据库迁移脚本正常执行（从旧版本升级）
- [ ] 开发环境和生产环境都能正常运行

**性能验收**:
- [ ] IPC 调用延迟 < 50ms（本地调用）
- [ ] 批量操作性能不低于当前实现
- [ ] 应用启动时间不增加超过 500ms

**安全验收**:
- [ ] Renderer Process 无法直接访问数据库文件
- [ ] IPC 接口有权限校验（防止恶意调用）
- [ ] 敏感数据（API Key）不在 IPC 日志中泄露

**代码质量验收**:
- [ ] 单元测试覆盖率 > 80%
- [ ] 所有文件符合代码规范（< 300 行）
- [ ] TypeScript 类型完整，无 any 类型
- [ ] 有完整的错误处理和日志记录

### 4.2 ContentPanel统一容器验收标准

**功能验收**:
- [ ] 支持多标签管理（新建、关闭、切换）
- [ ] 支持至少 3 种内容类型（文件树、飞书文档、设置）

- [ ] 标签拖拽排序

**用户体验验收**:

- [ ] 标签栏支持滚动（标签过多时）

**兼容性验收**:

- [ ] 飞书文档功能完全保留

**代码质量验收**:

- [ ] 类型定义完整
- [ ] 有单元测试和集成测试

## 5. 技术风险与依赖

### 5.1 数据库架构重构风险

**高风险**:
1. **数据迁移失败**: 用户升级后数据丢失
   - **缓解措施**: 迁移前自动备份数据库，失败时回滚

2. **IPC 性能问题**: 频繁的 IPC 调用导致性能下降
   - **缓解措施**: 批量操作、缓存策略、性能测试

**中风险**:
3. **类型不匹配**: IPC 传输的数据类型与预期不符
   - **缓解措施**: 使用 Zod 或 io-ts 进行运行时类型校验

4. **错误处理不完善**: IPC 调用失败时应用崩溃
   - **缓解措施**: 完善的错误处理和降级策略

### 5.2 ContentPanel统一容器风险

**高风险**:
1. **状态管理复杂**: 多标签状态难以维护
   - **缓解措施**: 使用成熟的状态管理库（Zustand）

2. **性能问题**: 多标签同时渲染导致卡顿
   - **缓解措施**: 内容懒加载、虚拟滚动

**中风险**:
3. **兼容性问题**: 现有组件迁移到新容器后功能异常
   - **缓解措施**: 充分的集成测试，逐步迁移

4. **用户体验问题**: 标签管理不符合用户习惯
   - **缓解措施**: 参考成熟产品（VS Code、Chrome）的交互设计
