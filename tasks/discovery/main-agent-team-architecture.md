## 当前理解

### 1. 职责边界

- Main Agent 应是唯一直接面向用户的逻辑主入口，负责理解目标、补全上下文、决定是否需要 Team、向用户发起确认、输出最终答复。
- Agent Team 在 MVP 中不应是独立一级入口，而应是 Main Agent 下的一种协作模式，用来做多角色拆解、任务委派、执行跟踪和结果汇总。
- Main Agent 对“用户关系”和“最终结果”负责；Agent Team 对“内部协作过程”和“阶段性交付”负责。
- Team 内部角色不直接接管用户会话，避免出现两个平级主控。

### 2. 共享能力

- Main Agent 和 Agent Team 共享同一套会话上下文、工作目录、Mind/Memory、知识检索、Skills、MCP、模型与工具运行时。
- Team 角色默认继承 Main Agent 的基础上下文与能力，不在 MVP 中引入独立长期记忆、独立全局配置、独立权限体系。
- 共享的是底层 runtime 和能力装配；差异主要体现在职责、流程和可见性，而不是另造一套底层引擎。

### 3. 激活方式

- 默认以单 Agent 模式进入，由 Main Agent 直接处理。
- 只有两种情况进入 Team 模式：用户显式要求多人协作，或 Main Agent 判断任务需要拆角色后提出建议。
- 无论哪种触发方式，切入 Team 前都应由 Main Agent 先给出一个清晰的 team plan，并等待用户确认。

### 3.1 渠道接入方式

- `唯一主入口` 应理解为“唯一逻辑入口”，而不是“只能有一个产品渠道”。
- Main Agent 应同时支持主聊天界面和 IM 渠道接入，例如飞书。
- 飞书、站内聊天等都应视为 Main Agent 的不同入口渠道，而不是不同的产品入口或不同的 Agent 体系。
- 因此 IM 对话能力应合并在这张联合架构卡里，而不是拆成另一个平级需求。

### 4. 确认点

- Main Agent 负责模式切换确认、team plan 确认、超出既定计划的新增委派确认、最终交付确认。
- Team 内部的常规步骤推进默认不逐步打断用户，否则会把多角色协作退化成频繁人工驱动。
- 高风险工具或敏感操作继续沿用现有运行时权限确认，不转嫁给 Team 角色私自决定。
- 用户应有权限查看 Team 结果面和 Team 会话；默认不是只允许系统内部可见。

### 5. 结果回填方式

- 子角色的执行结果先回填到 Team 上下文，形成可追踪的阶段产出。
- Main Agent 再把这些阶段产出压缩成主会话可读的进度更新、关键结论、产物引用和下一步建议。
- Team 应有独立结果面和可查看的 Team 会话，用来承载完整协作轨迹、阶段结果和最终沉淀。
- 但默认的对用户汇报路径仍应是 Main Agent 回到主会话做总结与交付。

### 6. 参考 OpenClaw 的启发

- OpenClaw 把 `default/main agent` 和其他 agent 区分开，并使用确定性的路由规则；对 Lumos 的启发是保留 Main Agent 作为默认主控，但不把 Agent Team 做成多个平级入口的路由系统。
- OpenClaw 的 sub-agent 默认隔离、默认限制递归深度，并有 allowlist、sandbox guard、timeout、archive 等保护；对 Lumos 的启发是 Team MVP 应是受控委派，而不是无限自治。
- OpenClaw 支持 thread-bound sub-agent session；对 Lumos 的启发是 Team 过程可以有独立上下文，但 follow-up 仍应绑定在当前主会话或当前 team run 之下，而不是裂变出多个最终结果面。

### 7. 执行闭环与异常处置

- Team 任务是否“执行完整”，不能靠角色自报完成，而应靠显式的 team run / task 状态推进来判断。
- 每个 task 在架构上都应至少有：负责人角色、输入、预期产出、依赖关系、当前状态、最近一次进展、失败原因或阻塞原因。
- Task 完成的判定应是“产出已返回且通过当前任务的完成检查”，而不是只看模型是否说“done”。
- Team Orchestrator 应持续检查是否仍有可运行任务；只有当全部任务进入终态，或进入明确阻塞态，team run 才结束。
- 异常不应静默吞掉；任何失败都必须转成可见状态：可重试、待人工确认、待依赖、已失败、已降级完成。
- 参考 OpenClaw 的 heartbeat / cron 思路，后台续跑应作为正式能力纳入：前台执行依赖事件驱动推进，后台执行依赖 scheduler/watchdog 定期唤醒待处理的 team run，保证任务稳定收口。

### 8. Agent Team 的管理方式

- 应先区分三个层级：`Role/Sub-agent Preset`、`Team Template`、`Team Run`。
- `Role/Sub-agent Preset` 表示单个执行角色的定义，例如职责、默认能力、输出要求。
- `Team Template` 表示一组角色及其默认协作关系。
- `Team Run` 表示某次任务里实际被实例化出来的团队执行实例。
- Main Agent 在运行时应动态创建 `Team Run`，而不是动态创造长期存在的 Team 定义。
- 用户侧可以预设 `Team Template`，用于指定常用角色组合、默认职责边界和协作风格；也可以预设少量通用 `Role/Sub-agent Preset`。
- 因此管理方式推荐采用混合模式：`Preset/Template` 可预设，`Run` 由 Main Agent 按当前任务动态实例化。
- 如果当前任务缺少合适角色，Main Agent 可以在当前 `Team Run` 内动态补一个临时角色，并默认沉淀为长期 `Role Preset`。
- 如果该角色只是某次任务的偶发特例，后续应允许用户再手动清理或合并，而不是要求每次创建前都确认是否保存。
- MVP 仍不建议支持 Main Agent 任意生成无限新角色体系；更稳妥的是优先复用内置或用户预设角色，不够时再补运行时角色并默认沉淀。
- 角色膨胀治理默认交给 Main Agent 处理：发现重复、低复用、职责重叠的角色时，由 Main Agent 负责建议合并、清理或重命名。

### 9. 工作协同机制

- Agent Team 的协同不应退化成固定流水线，也不应依赖完全无约束的自由聊天；更合适的是“分层 team + 结构化协作对象”。
- `Task` 是最小协作单元，负责定义做什么、谁做、依赖什么、产出什么。
- `Handoff` 是角色之间的交接包，至少包含目标、上下文、约束、预期输出、上游结果引用。
- `Artifact` 是执行产物，例如结论、文件引用、变更摘要、待确认项。
- `Status` 用来驱动推进，例如 `pending / ready / running / waiting / blocked / done / failed`。
- Team 内部协作应以“共享状态 + 结构化交接”为主，允许分层协作和并行推进。
- 更准确的协同关系应是：`Main Agent -> Team Orchestrator -> Domain Leads -> Workers`。
- `Main Agent` 负责面向用户，`Team Orchestrator` 负责团队调度，`Domain Leads` 负责领域拆解和小组管理，`Workers` 负责实际执行。
- 因此 Team 不只是一个流水线，而是一个由主控带领、可分层并行的专家协作队。
- 执行上应按真并行目标设计：多个 Leads/Workers 可以同时推进不同子任务，而不是只保留结构上的层级。

### 9.1 并行冲突与 Lock 原则

- 真并行执行必须配套资源级 lock，而不能只靠 session 级 lock。
- Lock 粒度建议至少覆盖：`workspace 子目录 / 文件集合 / 任务产物目标 / 外部资源句柄`。
- 一个 Worker 在进入执行前，应先声明本次要占用的资源范围；资源冲突时，由 Orchestrator 决定排队、改派、拆分或回退。
- 对同一资源的写操作默认互斥；读操作可并行，但一旦涉及修改、生成覆盖、重命名、发布等动作，必须获得独占 lock。
- Lock 不只是文件级；对浏览器会话、远端环境、第三方 API 目标、部署目标等共享资源，也应有同样的占用控制。
- 当任务异常退出、超时、AI 服务不可用或网络中断时，系统必须回收或续约失效 lock，避免 Team Run 被假死锁住。

### 10. 从 Claude Agent SDK 视角收敛

- Lumos 当前底座已经是 `@anthropic-ai/claude-agent-sdk` 的单 session 主链路：`query()` + `resume` + `permissionMode` + `tool_use/tool_result` + `hooks`。
- 因此联合架构的第一原则应是：继续复用 Claude SDK 的会话、权限、resume、hooks 能力，但不要把整个 Team 锁死在一个前台主会话里。
- 从 Claude 生态看，更合适的理解是：`Main Agent` 保持为主 chat session，而 `Team Run`、`Leads`、`Workers` 使用独立 Claude runs/sessions 或受控 subagents 承载执行。
- 协同的最小实现不必先落成复杂消息总线，而应先落成 Claude 生态已经能承载的四件事：主入口控制、独立执行 run、subagent 角色化、resume 续跑。
- 对 Lumos 来说，更自然的 MVP 是“Claude 主入口 + 分层 Team Run”，而不是“单会话串行分派”，也不是“完全自定义多 agent 平台”。

### 11. Team 形态判断

- 不建议把 Team 理解成固定顺序的流水线；更准确的形态应是“Main Agent 领导下的专家协作队”。
- Main Agent 更像 CEO 或统一入口，不直接等于每一步的执行者。
- Team Orchestrator 更像总负责人，负责带队执行而不是替 Main Agent 一直占住前台。
- Domain Leads 和 Workers 应有相对独立的职责、上下文和产出，不只是流水线里的步骤节点。
- Team 的推进应允许分支、回退、重做、评审驳回、改派和并行执行，而不是只能 A → B → C 单向前进。
- 也就是说，MVP 推荐的是“指挥式、分层并行的 team”，而不是“对等自治群聊”；但它也不应退化成死板流水线。

### 12. 主入口与执行占用分离

- `Main Agent 是唯一主入口` 不应被理解成 `Main Agent 必须持续占用同一个前台 Claude session 直到 Team 完成`。
- 在 Lumos 当前代码里，同一个 chat session 有运行锁和 `runtime_status`，因此如果把 Team 执行完全绑在主会话里，主会话会被长期占用。
- 更合理的联合架构应是：Main Agent 负责接收用户意图、生成计划、启动 Team Run、查询状态、汇总结果；Team Run 负责后台实际执行。
- 也就是说，`Main Agent 是控制面`，`Team Run 是执行面`，且 Team Run 内部还应支持 `Orchestrator -> Leads -> Workers` 的分层执行。
- 用户仍然只通过 Main Agent 发起、查看、干预 Team，但 Team 的长时间执行不应要求主会话一直保持 `running`。
- 当 Team Run 启动后，主会话应尽快回到可交互状态；后续由 Main Agent 通过“状态查询 / 结果通知 / 用户追问”重新接入该 Team Run。

### 12.1 渠道身份归一

- 渠道身份归一的意思是：同一个用户从站内聊天或飞书进入时，系统应尽量识别为同一个逻辑用户与同一套 Main Agent 体系，而不是两个互不相干的用户副本。
- 这不是新的产品入口问题，而是跨渠道会话归属和结果回看的连续性问题。
- 该问题目前可作为实现风险保留，不再作为阻塞性拍板项。

### 13. 产品体验目标

- 这套体验不应像“多 Agent 功能展示台”，而应像一个极强但克制的专业工作台。
- 用户第一感受应是：单入口清晰、复杂能力强、过程可见、结果可信，而不是界面复杂或概念过多。
- Team 模式应被感知为 Main Agent 能力自然升级，而不是突然跳进另一个产品。
- 复杂度应向内收敛：默认简单，按需展开；用户不必先理解整套团队编排概念才能用。
- 审美方向应追求成熟、克制、专业、精致，避免玩具化、多彩看板化、过度游戏化、多头像社交化。
- 同时这套体验应支持跨渠道一致性：无论用户从主聊天界面还是飞书进入，都应感觉是在和同一个 Main Agent 对话。

### 14. UI 结构建议

- 主会话仍是第一界面，也是默认停留界面。
- Team 模式启动后，建议出现一个独立但从属于主会话的 Team Workspace / 结果面，而不是完全跳转丢失主上下文。
- 推荐采用三层信息结构：
- `主会话层`：用户目标、Main Agent 汇报、关键确认、最终结果。
- `Team Workspace 层`：团队总览、阶段状态、负责人分布、当前阻塞、最终沉淀。
- `Team 会话层`：某个 Orchestrator / Lead / Worker 的具体执行记录，仅在用户下钻时查看。
- 主会话与 Team Workspace 应有明确主次：主会话负责“说给用户听”，Team Workspace 负责“把团队真相展开给用户看”。
- IM 渠道不需要完整承载 Team Workspace；IM 更适合作为轻量对话、提醒、确认和结果摘要入口。
- 完整 Team Workspace、深度下钻和多角色会话查看，仍应优先放在主聊天产品界面中。

### 15. UI 设计方向

- 视觉语言应偏“高端专业工作台”，而不是“聊天软件 + 若干 Agent 气泡”。
- 信息密度可以高，但必须有强层级：标题、阶段、状态、负责人、产物引用要一眼分开。
- 状态色应克制使用：重点依赖排版层级、间距、边界和轻量色彩，而不是大面积高饱和提示。
- Agent 呈现建议偏角色签名/职责标签，而非夸张头像和拟人卡通表达。
- Team Workspace 应更像“任务指挥台 + 研究工作台”的融合体，避免传统项目管理看板那种笨重感。
- 时间线、阶段切换、结果沉淀应有足够的留白与节奏感，让复杂过程看起来高级、安静、可信。

### 16. 交互设计原则

- 所有高价值交互都应遵循渐进披露：先给结论和下一步，再允许查看细节、过程、原始会话。
- 同一时刻只给用户一个主动作，避免出现多个相互竞争的 CTA。
- Main Agent 的汇报要面向决策，而不是面向日志；用户默认看到的是“现在发生了什么、我是否需要决定、结果是什么”。
- Team Workspace 的交互要面向理解与接管：用户能查看、暂停、追问、批准、回退，但不需要先学会复杂编排语言。
- 并行执行不能让界面显得吵闹；多个角色同时推进时，应优先汇总成阶段级进度，再允许下钻到单角色细节。
- 关键确认必须有明确上下文、风险说明和影响范围，避免用户在信息不完整时做授权。
- 最终结果不应只是一条聊天消息；应包含清晰结论、关键产物、团队贡献视图和可追溯入口。
- IM 渠道中的交互应更短、更明确、更偏行动导向：适合确认、催办、查看摘要，不适合承载完整复杂控制台。

### 18. IM / 飞书接入要求

- Main Agent 需要支持通过 IM 对话接入，例如飞书。
- 飞书应被定义为 Main Agent 的正式入口渠道之一，而不是临时桥接或边缘能力。
- 从飞书进入时，用户应仍然能够：
- 发起普通对话
- 触发 Team 模式
- 接收关键确认请求
- 查看阶段性汇报和最终结果摘要
- 但对于 Team Workspace、完整结果面、深度执行轨迹和多角色会话查看，推荐通过主产品界面承载，飞书只负责摘要与跳转。
- 因此推荐的多渠道关系是：
- `主产品界面`：完整控制台、完整结果面、完整 Team Workspace
- `飞书/IM`：轻入口、提醒、确认、摘要、跳转

### 19. 调度预算与默认约束

- 并行能力需要默认预算约束，而不是完全放开。
- 预算至少应包括：最大并行 Worker 数、单任务超时、单任务最大重试次数、单 Team Run 最大运行时长、模型/工具预算上限。
- 在没有用户显式调优前，系统应始终使用安全默认值，避免因为并行扩张导致成本失控或资源耗尽。
- 预算命中后，不应静默失败；应进入可见状态，由 Orchestrator 汇报当前受限原因。

### 17. 审美与品牌气质要求

- 产品气质应偏“顶级专业工具”，不是“炫技型 AI 实验品”。
- 设计应传达判断力、秩序感和控制感，让用户觉得自己在使用一个成熟的智能工作系统。
- 重点不是视觉花哨，而是高级的克制、精确的信息组织、稳健的动态反馈和统一的细节品味。
- 如果需要一个审美参照，方向应更接近“精密编辑器 + 高端研究控制台”，而不是“社交协作看板 + AI 头像广场”。

## 必须拍板的问题

- 当前无新增必须拍板问题；核心方向已收敛。

## 推荐关系

### 主入口

- 推荐 Main Agent 作为唯一主入口。
- 用户只和 Main Agent 对话，Main Agent 对外代表整个系统。
- 这个主入口可以通过多个渠道出现，例如主聊天界面和飞书，但它们在逻辑上都归属于同一个 Main Agent。

### Team 模式

- 推荐把 Agent Team 定义为 Main Agent 的 session-scoped 协作模式，而不是独立产品模块。
- Team 模式仅在复杂任务、多角色分工、需要阶段性交付时开启，默认关闭。
- 推荐把 Team 的“可管理对象”分成两层：可复用的 `Template` 和一次性的 `Run`。
- 推荐把 Team 体验定义成“主 Agent 带领一个临时组队的专家协作队”，而不是“进入另一个平行聊天入口”。
- Team Mode 进入后，用户感知上仍在 Main Agent 下，但执行上应启动独立的 Team Run，而不是把主会话一直锁死。
- 推荐把 Team Run 内部再拆成三层：`Orchestrator -> Domain Leads -> Workers`。
- Team Run 结束后，应留下一个独立 Team 结果面，供用户回看团队内部协作和完整交付。

### Team 管理

- 推荐采用混合模式，而不是“全动态”或“全预设”。
- 系统应提供少量内置 Team Template，覆盖通用协作场景。
- 用户可以选择或维护自己的 Team Template，但这属于配置层，不等于每次都要手动搭 Team。
- 如果需要更细粒度复用，可以有少量 `Role/Sub-agent Preset`；但用户侧主要管理对象仍应是 `Team Template`，避免配置层过碎。
- Main Agent 在具体任务里只负责选择合适模板，或从模板裁剪出本次需要的角色集合，然后创建 Team Run。
- 当模板不完全匹配时，Main Agent 可以在本次 Team Run 里补一个角色完成缺口，并默认沉淀为长期 `Role Preset`。
- 如果当前任务不适合任何模板，Main Agent 可以退回单 Agent 处理，而不是强行造一个临时 Team。

### 委派机制

- 推荐由 Main Agent 先产出结构化 team plan，明确任务、负责人角色、预期产出、依赖关系。
- 用户确认后，再由 Main Agent 启动独立 Team Run，并把执行权交给 Team Orchestrator。
- Team Orchestrator 按 plan 真并行拉起一个或多个 Domain Leads，再由每个 Lead 视需要组织自己的 Workers。
- 参考 OpenClaw 的受控层级思路，MVP 建议只允许有限层级：`Main Agent -> Orchestrator -> Leads -> Workers`，不做无限递归扩编。
- 新增超范围委派或计划变更时，回到 Main Agent 发起确认。
- 委派后的推进不是固定线性流程；Orchestrator 或 Leads 可以根据阶段结果决定继续、回退、改派、追加评审或结束。
- 这类推进不要求一直占用前台 Main Session；更合理的是由独立 Team Run 持续推进，Main Agent 只在关键节点重新介入。

### 回填机制

- 推荐采用“Workers -> Leads -> Orchestrator -> Team 结果面 -> Main Agent -> 主会话”的分层回填。
- Team 结果面负责保留完整团队过程、阶段结论、最终沉淀和可查看的 Team 会话。
- 主会话负责默认汇报：进度摘要、关键结论、文件/产物引用和最终答复。
- IM 渠道负责轻量回填：提醒、关键确认、阶段摘要、最终摘要，以及必要时跳转到完整 Team 结果面。
- 因此产品上应同时存在两个层次的结果：
- `主会话结果`：默认用户入口看到的汇报结果
- `Team 结果面`：用户可主动进入查看的完整团队执行结果

### 产品设计

- 推荐把主体验定义为“一个入口、两个观察层级、多个执行主体”。
- 用户始终先从 Main Agent 获得方向感，再决定是否进入 Team Workspace 查看真并行团队的完整工作面。
- Team 能力应让用户感觉自己像在管理一支顶级团队，而不是手动操作一堆 agent。

### UI 设计

- 推荐主会话保持简洁、稳定、强可读；Team Workspace 则承担高密度信息展示。
- Team Workspace 默认看总览，不默认打开所有细节；细节会话和执行日志必须可下钻，但不应默认铺满。
- 视觉上应强调“阶段、负责人、结果、风险、下一步”五个锚点。

### 交互设计

- 推荐把 Team 的默认交互设计成“汇报式”：Main Agent 主动汇报，用户按需展开。
- 推荐把 Team Workspace 的交互设计成“控制台式”：可查看、可追问、可批准、可介入、可回看。
- 推荐把单个 Worker/Lead 会话设计成“证据式”：用于验证团队内部确实做了什么，而不是默认用户阅读面。

### 协同技术

- MVP 推荐直接贴着 Claude Agent SDK 能力做，而不是一开始就引入独立消息总线或复杂分布式编排。
- 具体建议是：
- `Main Agent = 一个主控制会话`
- `Team Run = 一个独立团队执行实例`
- `Orchestrator / Leads / Workers = 一个或多个独立 Claude 执行 run/session`
- `委派 = Main Agent 启动 Team Run，再由 Orchestrator / Leads 组织多层真并行委派`
- `角色定义 = Claude subagent prompt / preset`
- `延续执行 = 各执行 run 各自复用 SDK session resume`
- `高风险控制 = 继续复用现有 permissionMode / canUseTool`
- `校验与收口 = 用 hooks 和主会话后处理做检查`
- `后台稳定性 = 纳入 watchdog / scheduler，定期续跑、补偿、检查未收口任务`
- 因此从技术路线看，首版更像“Claude 主控制会话 + 分层并行 Team Run”的指挥式 team，不是“自研自治型多 Agent 平台”，也不是“固定死板流水线”。

### 执行保证

- 推荐把“保证执行完整”定义成：所有 task 都进入明确终态，且成功 task 都附带预期产出或可验证引用。
- Orchestrator / Leads 在每轮执行后都要做一次状态归并：继续下一个任务、等待依赖、请求确认、或判定失败。
- 对长任务、异步任务或外部等待，推荐保留 `running` / `waiting` / `blocked` 等中间态，而不是假装一次对话内必须做完。
- 默认原则应是“能继续就继续跑”；只要任务还符合既定目标和预期流程，就不应因为普通中断而永久停下。

### 异常处置

- 工具失败、模型输出不合格、依赖缺失、权限未获批、超时无进展，都应被视为结构化异常，而不是普通聊天文本。
- 可恢复异常优先重试或回退到同级替代步骤；不可恢复异常回到 Main Agent 发起用户确认。
- 需要使用单独的 team-run watchdog 定期检查“超时未推进”或“等待条件已满足”的任务，再决定是否唤醒继续执行，确保所有任务稳定完成。
- Watchdog 的停止条件应严格限定为：任务在仍然遵循既定目标和预期流程的前提下，已经无法继续推进，此时才停下来等待用户。
- 除此之外，任务不应因为临时故障而直接停住；网络抖动、终端异常、AI 服务暂时不可用等情况，默认都应进入自动恢复、重试或补偿流程。
- 因此默认策略是：
- `不可继续`：等待用户
- `可恢复`：自动继续跑
- `不确定是否越界`：回 Main Agent 发起确认
