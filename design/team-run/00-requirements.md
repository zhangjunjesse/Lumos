- 无 Agent 实例化逻辑
- 无任务调度器
- 无 Agent 间通信机制
- 无依赖解析和并行执行

## 2. 功能需求

### 2.1 核心功能

#### F1: Team Run 生命周期管理

**描述**: 管理 Team Run 从创建到完成的完整生命周期

**状态流转**:
```
pending → ready → running → [waiting/blocked] → done/failed
```

**关键操作**:
- 创建 Team Run（从 TeamPlan 初始化）
- 启动执行
- 暂停/恢复
- 取消执行
- 标记完成/失败

#### F2: 任务依赖解析

**描述**: 解析 TeamRunStage 之间的依赖关系，构建执行图

**输入**:
```typescript
phases: TeamRunStage[] = [
  { id: 'phase-1', dependsOn: [] },
  { id: 'phase-2', dependsOn: ['phase-1'] },
  { id: 'phase-3', dependsOn: ['phase-1'] },
  { id: 'phase-4', dependsOn: ['phase-2', 'phase-3'] }
]
```

**输出**:
```typescript
batches: string[][] = [
  ['phase-1'],           // 批次 1：无依赖
  ['phase-2', 'phase-3'], // 批次 2：依赖 phase-1
  ['phase-4']            // 批次 3：依赖 phase-2 和 phase-3
]
```

**约束**:
- 检测循环依赖并拒绝执行
- 支持 DAG（有向无环图）
- 依赖的任务必须全部成功才能开始

#### F3: 并行任务调度

**描述**: 在满足依赖的前提下，并行执行多个任务

**配置**:
```typescript
budget: {
  maxParallelWorkers: 3,  // 最大并行数
  maxRetriesPerTask: 1,   // 每个任务最大重试次数
  maxRunMinutes: 120      // 最大执行时间（分钟）
}
```

**行为**:
- 同一批次的任务并行执行
- 并行数不超过 `maxParallelWorkers`
- 任务失败时根据 `maxRetriesPerTask` 决定是否重试
- 超过 `maxRunMinutes` 自动取消

#### F4: Agent 实例化与执行

**描述**: 根据 TeamPlanRole 创建 Agent 实例并执行任务

**流程**:
1. 根据 `ownerRoleId` 查找对应的 TeamPlanRole
2. 加载 Role 的 systemPrompt（可能来自 AgentPreset）
3. 创建独立的 Claude SDK session
4. 执行任务并捕获输出
5. 更新 TeamRunStage 状态和结果

**隔离要求**:
- 每个 Agent 使用独立的 SDK session
- 工作目录隔离（可选）
- 环境变量隔离

#### F5: Agent 间通信

**描述**: 支持 Agent 之间传递上下文和结果

**通信模式**:
- **依赖传递**: 后续任务可以访问依赖任务的输出
- **共享上下文**: 所有 Agent 可以访问 TeamRun.context
- **消息传递**: Agent 可以向其他 Agent 发送消息（未来扩展）

**数据流**:
```typescript
// Phase 1 输出
phase1.latestResult = "架构设计完成，使用 MVC 模式"

// Phase 2 可以访问
const dependencies = await getDependencyResults(['phase-1'])
// dependencies = [{ id: 'phase-1', result: "架构设计完成..." }]
```

#### F6: 状态同步与监控

**描述**: 实时更新 TeamRun 和 TeamRunStage 状态，供 UI 展示

**更新时机**:
- 任务开始时：`status = 'running'`
- 任务完成时：`status = 'done'`, 更新 `latestResult`
- 任务失败时：`status = 'failed'`, 记录错误
- 任务等待时：`status = 'waiting'`（依赖未完成）
- 任务阻塞时：`status = 'blocked'`（依赖失败）

**持久化**:
- 每次状态变化立即写入数据库
- 更新 `updatedAt` 时间戳
- 更新 TeamRun 的 `lastUpdatedAt`

#### F7: 错误处理与恢复

**描述**: 处理执行过程中的各种错误情况

**错误类型**:
- **任务失败**: Agent 执行出错
- **超时**: 超过 `maxRunMinutes`
- **依赖失败**: 依赖的任务失败
- **系统错误**: 数据库、网络等基础设施错误

**恢复策略**:
- 任务失败：根据 `maxRetriesPerTask` 重试
- 超时：标记为 failed，记录超时原因
- 依赖失败：标记为 blocked，不执行
- 系统错误：记录错误，标记 TeamRun 为 failed

**恢复机制**:
- 支持从失败点恢复（`resumeRun`）
- 重置失败的任务状态为 ready
- 重新执行失败的任务

**NFR-6: 可恢复性**
- 应用重启后可以恢复未完成的 Team Run
- 支持断点续传
- 保留完整的执行日志

### 3.3 可维护性要求

**NFR-7: 代码质量**
- 单文件不超过 300 行

- 测试覆盖率 > 80%

**NFR-8: 可观测性**
- 记录关键操作日志

## 4. 技术挑战

### 4.1 并发控制

**挑战**: 如何在满足依赖的前提下，最大化并行执行效率？

**难点**:
- 动态调度：任务完成后立即启动下一批
- 资源限制：不超过 `maxParallelWorkers`
- 优先级：如何决定先执行哪个任务？

**候选方案**:
1. **简单队列**: 按批次顺序执行，批次内并行
2. **优先级队列**: 根据依赖深度、预估时间等排序
3. **工作窃取**: 空闲 Worker 主动拉取任务

### 4.2 状态管理

**挑战**: 如何保证多个 Agent 并发执行时的状态一致性？

**难点**:
- 竞态条件：多个 Agent 同时更新 TeamRun
- 死锁：循环依赖导致互相等待
- 状态同步：UI 需要实时看到最新状态

**候选方案**:
1. **乐观锁**: 使用版本号，更新时检查冲突
2. **悲观锁**: 使用 `SELECT ... FOR UPDATE`
3. **事件驱动**: 状态变化发送事件，统一处理

### 4.3 Agent 通信

**挑战**: 如何让 Agent 之间高效、安全地传递信息？

**难点**:
- 数据量：输出可能很大（代码、文档）
- 格式：如何结构化输出？
- 安全：如何防止恶意注入？

**候选方案**:
1. **数据库共享**: 通过 `latestResult` 字段传递
2. **消息队列**: 使用 Redis/RabbitMQ
3. **文件系统**: 写入临时文件，传递路径

### 4.4 错误处理

**挑战**: 如何优雅地处理各种错误情况？

**难点**:
- 部分失败：一个任务失败，其他任务如何处理？
- 级联失败：依赖失败导致后续任务全部失败
- 恢复策略：重试？跳过？人工介入？

**候选方案**:
1. **快速失败**: 任何错误立即停止整个 Team Run
2. **最大努力**: 尽可能执行，记录失败任务
3. **人工介入**: 失败时暂停，等待用户决策

### 4.5 资源隔离

**挑战**: 如何隔离不同 Agent 的执行环境？

**难点**:
- SDK session 隔离：避免上下文混淆
- 文件系统隔离：避免文件冲突
- 资源限制：防止单个 Agent 占用过多资源

**候选方案**:
1. **进程隔离**: 每个 Agent 独立进程
2. **Session 隔离**: 共享进程，独立 SDK session
3. **容器隔离**: 使用 Docker（过重）

## 5. 约束条件

### 5.1 技术约束

**C-1: 技术栈限制**
- 必须使用 Claude Agent SDK
- 必须使用 better-sqlite3（不支持真正的并发写入）
- 必须在 Electron 主进程或 Next.js API 路由中运行

**C-2: SDK 限制**
- Claude SDK 不支持真正的多 Agent 协作
- 每个 session 是独立的，无法直接通信
- SDK 的 permission mode 可能影响执行

**C-3: 数据库限制**
- SQLite 不支持真正的并发写入
- 需要使用 WAL 模式提升并发性能
- 事务嵌套有限制

### 5.2 业务约束

**C-4: 用户体验约束**
- 启动 Team Run 必须在 1 秒内响应
- 状态更新延迟不超过 2 秒
- 必须支持取消正在执行的 Team Run

**C-5: 成本约束**
- 每个 Team Run 的 API 调用成本应可预估
- 支持设置预算上限（token 数、金额）
- 超出预算自动停止

**C-6: 安全约束**
- Agent 不能访问其他 Agent 的敏感数据
- 用户可以审查 Agent 的所有操作
- 支持沙箱模式（只读文件系统）

## 6. 关键问题

### Q1: 执行模型选择

**问题**: 应该使用什么执行模型？

**选项**:
- **A. 顺序执行**: 简单但慢，一个任务完成后才开始下一个
- **B. 批次并行**: 按依赖分批，批次内并行
- **C. 完全并行**: 只要依赖满足就立即执行，最大化并行度

**权衡**:
| 选项 | 优点 | 缺点 | 复杂度 |
|------|------|------|--------|
| A | 实现简单，易调试 | 效率低，用户等待时间长 | 低 |
| B | 平衡效率和复杂度 | 批次边界可能不是最优 | 中 |
| C | 效率最高 | 实现复杂，调试困难 | 高 |

**建议**: 先实现 B（批次并行），未来优化为 C

### Q2: Agent 实例化方式

**问题**: 如何创建和管理 Agent 实例？

**选项**:
- **A. 每次创建新 session**: 每个任务都创建新的 SDK session
- **B. 复用 session**: 同一个 Role 的任务复用 session
- **C. 长连接 session**: session 在整个 Team Run 期间保持

**权衡**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A | 隔离性好，无状态污染 | 启动慢，无上下文 |
| B | 有上下文，效率高 | 可能状态污染 |
| C | 最快，上下文最完整 | 资源占用高 |

**建议**: 使用 A（每次新建），保证隔离性

### Q3: 状态持久化策略

**问题**: 何时将状态写入数据库？

**选项**:
- **A. 实时写入**: 每次状态变化立即写入
- **B. 批量写入**: 积累多个变化后一次性写入
- **C. 异步写入**: 状态变化放入队列，后台写入

**权衡**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A | 数据最新，崩溃不丢失 | 写入频繁，性能差 |
| B | 性能好 | 崩溃可能丢失数据 |
| C | 性能最好，不阻塞 | 实现复杂，数据延迟 |

**建议**: 使用 A（实时写入），配合 WAL 模式优化性能

### Q4: 错误恢复策略

**问题**: 任务失败后如何处理？

**选项**:
- **A. 快速失败**: 任何失败立即停止整个 Team Run
- **B. 继续执行**: 失败任务标记为 failed，其他任务继续
- **C. 用户决策**: 失败时暂停，等待用户选择

**权衡**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A | 简单，避免浪费资源 | 可能过于激进 |
| B | 最大化完成任务数 | 可能浪费资源 |
| C | 灵活，用户控制 | 需要人工介入 |

**建议**: 使用 B（继续执行），配合重试机制

### Q5: 通信机制设计

**问题**: Agent 之间如何传递数据？

**选项**:
- **A. 数据库字段**: 通过 `latestResult` 传递
- **B. 文件系统**: 写入临时文件
- **C. 消息队列**: 使用 Redis/内存队列

**权衡**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A | 简单，无额外依赖 | 大数据存储效率低 |
| B | 支持大文件 | 需要清理，路径管理复杂 |
| C | 高效，解耦 | 增加依赖，复杂度高 |

**建议**: 使用 A（数据库字段），限制输出大小（如 10KB）

## 7. 设计原则

### P1: 简单优先
- 先实现最简单可用的版本
- 避免过度设计
- 优先满足核心需求

### P2: 渐进增强
- 第一版：顺序执行 + 基础错误处理
- 第二版：批次并行 + 重试机制
- 第三版：完全并行 + 高级调度

### P3: 可观测性
- 记录所有关键操作
- 提供详细的执行日志
- 支持调试模式

### P4: 容错设计
- 假设任何操作都可能失败
- 提供恢复机制
- 避免级联失败
