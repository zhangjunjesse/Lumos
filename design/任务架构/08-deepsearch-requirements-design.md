# DeepSearch 需求设计文档

## 0. 编号结论

本方案建议新增 `08` 主题。

其中主文档为：

- `08-deepsearch-requirements-design.md`
- `08-deepsearch-architecture-design.md`

补充文档可继续围绕：

- `bb-browser` 结合方式
- 部署与本地使用形态
- Phase 1 实现拆解
- UI 与交互
- 数据与 API 合同
- 工程落地拆解

原因：

- 它不是单纯的浏览器能力，不应只并入现有 Browser Workspace 说明
- 它不是单纯的调度问题，不应直接塞进 `03`
- 它不是单纯的执行步骤问题，不应直接塞进 `05`
- 它既要服务用户手动使用，也要服务对话和工作流调用
- 它必须先作为独立模块完成可用性验证，再由聊天与工作流复用

因此 `08` 的产品需求和架构设计应单独成文，并允许继续追加补充设计文档。

## 1. 问题背景

用户当前希望在 Lumos 中内置一个真正可用的 DeepSearch 能力，重点场景包括：

- 对反爬站点进行深度研究，而不是只做浅层搜索结果抓取
- 优先复用 Lumos 内置浏览器的共享登录态，减少登录墙和 Cookie 鉴权带来的阻塞
- 允许用户自己先在产品界面完成登录、配置和验证，再让 LLM 在对话或工作流里调用
- 让用户可以回看每次抓取的证据、正文、截图和失败原因，而不是只拿到一段最终摘要

当前 Lumos 已具备的基础：

- Electron 内置浏览器
- 共享浏览器 Session / Cookie
- Browser bridge / Chrome DevTools MCP
- 聊天调用工具和工作流能力调用的主链

但当前仍缺少一个面向产品的、独立可用的 DeepSearch 模块。

---

## 2. 产品定位

DeepSearch 的产品定位不是“另一个 browser step”，而是：

- 一个**可独立使用**的产品模块
- 一个**基于共享登录态**的深度研究执行器
- 一个**可被 LLM 调用**的高层工具
- 一个**可被 Workflow 复用**的能力服务

它要解决的核心问题是：

- 用户如何先把站点登录态准备好
- 系统如何基于这些登录态执行可恢复的深度抓取
- 用户和 LLM 如何查看、消费和复用抓取结果

---

## 3. 核心设计判断

### 3.1 先独立，再接入

DeepSearch 必须先做成独立模块，而不是一开始直接耦合进 Workflow。

原因：

- 用户会在对话外直接使用它
- 需要独立验证登录态、抓取质量、结果展示、失败恢复
- 如果先强耦合到 Workflow，会导致问题定位、可验收性和复用性都变差

### 3.2 登录态不是增强项，而是前置主链

如果目标是用共享登录态解决反爬问题，那么站点登录态管理必须作为 Phase 1 核心主链。

没有以下能力，DeepSearch 不可用：

- 检测站点是否已登录
- 引导用户登录
- 判断 Cookie / Session 是否过期
- 登录成功后自动恢复执行

### 3.3 给 LLM 的应该是高层能力，不是底层浏览器脚本

LLM 不应直接操纵 `BrowserManager` 或直接拼接底层浏览器动作。

LLM 使用 DeepSearch 的方式应该是：

- 调用高层 `deepsearch` 工具
- 传入业务语义参数
- 接收结构化运行状态和结果
- 在登录态缺失时收到明确的 `waiting_login` 反馈

---

## 4. 目标与非目标

### 4.1 核心目标

- 在左侧侧边栏 `扩展` 下新增正式 `DeepSearch` 页签
- 让用户可以查看和配置各站点登录态状态
- 让用户可以主动发起、暂停、恢复和查看 DeepSearch 任务
- 让用户可以浏览所有历史抓取记录、证据和详细内容
- 让对话和工作流可以通过统一能力入口调用 DeepSearch
- 在遇到登录失效、页面受限、抓取不完整时，给出可恢复、可追踪的状态反馈

### 4.2 非目标

Phase 1 不追求以下内容：

- 不做一个覆盖所有网站的通用反爬平台
- 不承诺自动绕过验证码、IP 风控、设备指纹风控
- 不允许 LLM 直接执行任意浏览器脚本作为正式主链
- 不把 DeepSearch 直接下沉成 Workflow DSL 的基础 step type
- 不要求一开始就提供复杂的多租户账号管理

---

## 5. 产品形态

## 5.1 入口位置

正式入口放在：

- 左侧侧边栏
- `扩展`
- `DeepSearch`

这是一个正式产品入口，不依赖测试页，不隐藏在聊天内部，也不强迫用户先理解工作流。

## 5.2 页面目标

`DeepSearch` 页面至少要承担三类事情：

1. 站点连接管理
2. 抓取运行与历史查看
3. 抓取内容和证据查看

---

## 6. 页面信息架构

Phase 1 建议 `DeepSearch` 页面采用单页四区结构（顶部任务区 + 左中右三栏），而不是一开始拆很多子路由。

### 6.1 顶部区：任务发起与运行状态

用户可直接：

- 输入查询词 / 研究目标
- 选择站点范围
- 选择页面来源模式
  - 接管当前活动页
  - 新建受管页
- 选择最大抓取页数 / 深度
- 选择执行严格度
  - `strict`
  - `best_effort`
- 启动 DeepSearch

同时显示当前运行中的任务状态：

- 等待开始
- 运行中
- 等待登录
- 已暂停
- 已完成
- 部分完成
- 失败
- 已取消

### 6.2 左栏：站点连接与登录态

至少支持：

- 站点列表
- 当前登录状态
- 最近验证时间
- 失效/异常提示
- “去登录”按钮
- “重新检查”按钮

站点状态建议统一为：

- `未连接`
- `已连接`
- `疑似过期`
- `已失效`
- `检查失败`

### 6.3 中栏：抓取记录列表

展示所有 DeepSearch 历史运行记录，包括：

- 查询词 / 任务标题
- 站点范围
- 创建时间
- 完成时间
- 当前状态
- 抓取页数
- 证据数
- 是否因登录态阻塞

### 6.4 右栏：详细内容与证据

用户点击一条记录后，可以查看：

- 运行摘要
- 结构化结果
- 抓取到的页面正文
- 关键片段
- 页面截图
- 原始 URL
- 失败原因 / 阻塞原因

## 7. 功能需求

## 7.1 站点连接管理

系统必须支持：

- 检查目标站点在共享浏览器上下文中的登录态
- 展示站点连接状态
- 支持一键打开该站点登录页
- 支持登录后刷新状态
- 支持标记该站点是否为 DeepSearch 必须登录站点

系统不要求：

- 用户手动复制 Cookie
- 用户理解底层 Session Partition
- 用户自己处理登录恢复时的技术细节

## 7.2 DeepSearch 任务发起

用户至少可以配置：

- 查询词 / 研究任务描述
- 站点范围
- 页面来源模式
  - `接管当前活动页`
  - `新建受管页`
- 研究目标
  - 结果浏览
  - 证据收集
  - 正文抓取
  - 研究摘要
- 最大抓取页数
- 最大深度
- 执行严格度
  - `strict`
  - `best_effort`
- 是否保留截图和正文证据

### 7.2.1 页面来源模式

DeepSearch 必须正式支持两种页面来源模式：

- `接管当前活动页`
  - 允许系统正式接管用户当前浏览器中的活动页
  - 适用于用户已经手动登录、已经打开目标页面、希望沿当前上下文继续深挖的场景
- `新建受管页`
  - 由 DeepSearch 在共享浏览器上下文中新建并管理页面
  - 适用于需要稳定批量抓取、减少打扰用户当前浏览过程的场景

但无论采用哪种模式，都必须满足：

- 进入正式执行前显式绑定 `runId + pageId`
- 不允许按域名隐式挑一个 tab 就执行
- 被接管的活动页一旦绑定到某次 run，其页面事件、证据和失败原因都必须可追溯到该 run

### 7.2.2 执行严格度

DeepSearch 必须把“页面来源模式”和“执行严格度”分成两个独立开关，而不是混成一个模糊选项。

`strict` 模式下：

- 关键站点登录失效时，任务进入 `等待登录`
- 关键站点执行失败时，任务直接失败
- 不应把缺页、缺正文或关键证据缺失包装成成功

`best_effort` 模式下：

- 能跑的站点先继续执行
- 被登录阻塞、风控阻塞或页面失败的站点记录为未完成
- 只要目标站点中有一部分未完成，最终状态就应返回 `部分完成`

这意味着：

- `partial` 不是异常兜底文案
- 它是 `best_effort` 模式下的正式结果状态

## 7.3 登录阻塞与恢复

当任务需要登录态但当前不可用时，系统必须：

- 返回明确的 `等待登录` 状态
- 告诉用户是哪个站点阻塞
- 提供一键去登录入口
- 登录完成后允许恢复原任务，而不是要求全部重建

并且要明确区分两种行为：

- `strict`
  - 关键站点一旦阻塞，整个 run 进入 `等待登录`
- `best_effort`
  - 非关键站点阻塞时，run 可以继续跑其他站点
  - 最终按 `部分完成` 收口

## 7.4 抓取执行与证据留存

每次任务至少要产出：

- 任务级摘要
- 页面级结果列表
- 每页的标题、URL、抓取时间
- 文本摘录
- 必要时的截图
- 页面来源站点
- 是否成功抓到正文

如果抓取失败，也必须保留：

- 失败页面 URL
- 失败阶段
- 错误原因
- 是否与登录态有关

如果任务以 `best_effort` 结束，也必须保留：

- 哪些站点成功完成
- 哪些站点被跳过或阻塞
- `partial` 的直接原因
- 每个失败页对应的证据或错误摘录

## 7.5 历史记录与详情查看

用户必须可以：

- 查看所有历史抓取任务
- 查看单次任务的完整细节
- 打开单条抓取结果的正文与证据
- 复用历史任务参数重新运行

## 7.6 对话与工作流复用

虽然 DeepSearch 本体先独立，但系统必须从一开始就按“可复用”设计。

需要支持的消费方式：

- 聊天中的 LLM 调用
- Workflow 中的能力调用
- 后续可能的 MCP facade

但这些调用都不应绕过独立模块本身的状态管理、登录管理和结果存储。

---

## 8. LLM 可调用需求

给 LLM 的不是底层浏览器命令，而是高层 DeepSearch 工具。

LLM 调用时，系统至少要支持以下状态：

- `pending`
- `running`
- `waiting_login`
- `paused`
- `completed`
- `partial`
- `failed`
- `cancelled`

其中：

- `waiting_login` 是第一类正式状态，不是普通错误
- `resume` 是动作，不是单独状态

LLM 侧建议暴露以下高层工具：

- `deepsearch.start`
- `deepsearch.get_result`
- `deepsearch.pause`
- `deepsearch.resume`
- `deepsearch.cancel`

LLM 需要能拿到：

- `runId`
- 当前状态
- 用户可执行动作
- 最终摘要
- 证据列表
- 失败原因

---

## 9. UI 验收口径

Phase 1 从 UI 上至少要可验收这些能力：

- 左侧侧边栏可以进入正式 `DeepSearch` 页面
- 页面里能看到各站点登录态状态
- 未登录时可一键去登录
- 登录后可重新检查状态
- 能发起 DeepSearch 任务
- 能选择“接管当前活动页”或“新建受管页”
- 能选择 `strict` 或 `best_effort`
- 任务阻塞时能看到“等待登录”
- `best_effort` 任务未全部完成时能看到“部分完成”
- 能查看历史抓取记录
- 能打开某次任务的详细内容、正文和截图

用户在 Phase 1 不一定能验收：

- 所有反爬站点都能稳定完整抓取
- 所有网站都能自动恢复登录
- 复杂验证码和 IP 风控的自动化处理

---

## 10. Phase 划分

### Phase 1：独立模块最小可用

- 正式 `DeepSearch` 页面入口
- 站点登录态检查与登录引导
- 单任务 DeepSearch 执行
- `接管当前活动页 / 新建受管页` 两种模式
- `strict / best_effort` 两种执行语义
- 历史记录与详情查看
- 聊天可调用的高层 tool 契约

### Phase 2：结果质量增强与多入口复用

- 更多站点策略
- 更稳定的正文提取
- 更清晰的证据包和失败分类
- 结果去重与多页汇总增强
- Workflow 复用同一服务

### Phase 3：外部形态与治理增强

- 独立 MCP facade
- 更完整的权限、配额、治理与审计

---

## 11. 当前结论

DeepSearch 在 Lumos 中的正确产品落点是：

- 先做正式独立入口
- 先把登录态管理主链做出来
- 先让用户可以单独使用并验收
- 再让聊天和 Workflow 复用

如果跳过这一步，直接把它塞进 Workflow 或聊天主链，最终会得到一个难调试、难验收、难恢复的能力，而不是一个真正可用的 DeepSearch 模块。
