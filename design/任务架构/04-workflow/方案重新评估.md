# 工作流方案重新评估

## 背景

经过深入调研，发现之前的 Temporalite 方案不可行：
- Temporalite 已于 2024年4月归档，不再维护
- 功能虽然合并到 Temporal CLI，但仍标注为"开发用途"
- 不适合生产环境

## 三个可行方案对比

### 方案A：完整 Temporal Server

**架构**：
```
Lumos (Electron)
  ├─ Temporal Server (独立进程, Go)
  │   └─ SQLite 存储
  ├─ Temporal Worker (Node.js)
  └─ UI 层
```

**优点**：
- 成熟稳定，生产级别
- 功能完整（状态管理、错误恢复、长时间运行）
- 社区支持好

**缺点**：
- 需要管理独立进程（启动/停止/崩溃恢复）
- 资源占用较大（估计 100-200MB）
- 增加应用复杂度
- 用户可能不希望后台常驻多个进程

**适用场景**：
- 需要复杂工作流（长时间运行、复杂状态管理）
- 用户能接受后台进程
- 对稳定性要求高

---

### 方案B：自建轻量级工作流引擎

**架构**：
```typescript
// 纯 TypeScript 实现
class WorkflowEngine {
  async execute(workflow: WorkflowDSL) {
    // 解析 DAG
    // 执行节点
    // 状态持久化到 SQLite
  }
}
```

**优点**：
- 轻量级（< 10MB 内存）
- 无需额外进程
- 完全可控，易于定制

**缺点**：
- **开发成本高**（需要实现状态管理、错误恢复、并发控制）
- **稳定性风险**（需要大量测试）
- **维护成本高**（需要持续优化）
- 功能可能不如 Temporal 完善

**需要实现的核心功能**：
1. DAG 解析和拓扑排序
2. 节点执行引擎（顺序/并行/条件/循环）
3. 状态持久化（SQLite）
4. 错误处理和重试
5. 暂停/恢复/取消
6. 长时间运行支持
7. 并发控制
8. 死锁检测

**预估工作量**：
- 核心引擎：3-4 周
- 节点类型：2 周
- 错误处理和恢复：2 周
- 测试和优化：2-3 周
- **总计：9-11 周**（且可能低估）

**适用场景**：
- 只需要简单工作流
- 有足够的开发时间和测试资源
- 团队有工作流引擎开发经验

---

### 方案C：简化版任务编排（推荐）

**核心思路**：不做通用工作流引擎，只做任务编排

**架构**：
```typescript
// 简化的任务编排
interface TaskPlan {
  steps: TaskStep[];  // 只支持顺序执行
  onError?: 'retry' | 'fail' | 'continue';
}

interface TaskStep {
  type: 'agent' | 'browser' | 'notification';
  config: any;
  condition?: string;  // 简单的条件判断
}

class TaskExecutor {
  async execute(plan: TaskPlan) {
    for (const step of plan.steps) {
      // 检查条件
      if (step.condition && !evalCondition(step.condition)) {
        continue;
      }

      // 执行步骤
      try {
        await executeStep(step);
      } catch (error) {
        if (plan.onError === 'retry') {
          await retry(step);
        } else if (plan.onError === 'fail') {
          throw error;
        }
        // continue: 忽略错误继续
      }
    }
  }
}
```

**支持的功能**：
- ✅ 顺序执行多个步骤
- ✅ 简单条件分支（if/else）
- ✅ 错误重试
- ✅ 状态持久化
- ✅ 暂停/恢复/取消
- ❌ 复杂 DAG（并行、循环、子工作流）
- ❌ 长时间运行（> 1小时）
- ❌ 分布式执行

**优点**：
- **开发成本低**（1-2 周）
- **足够简单**，易于理解和维护
- **满足 80% 的需求**（大部分任务是顺序执行）
- 无需额外进程

**缺点**：
- 功能受限，不支持复杂工作流
- 如果未来需要复杂工作流，需要重构

**渐进式升级路径**：
1. **Phase 1**：实现简化版（1-2 周）
2. **验证需求**：观察用户实际使用情况
3. **Phase 2**：如果确实需要复杂工作流，再考虑：
   - 升级到方案A（Temporal）
   - 或扩展方案C（增加并行、循环等）

**适用场景**：
- 大部分任务是顺序执行
- 不需要复杂的并行和循环
- 希望快速上线验证需求

---

## 推荐决策

**我推荐方案C（简化版任务编排）**，理由：

1. **需求不明确**：我们还不知道用户真正需要什么样的工作流
2. **快速验证**：1-2 周就能上线，快速验证需求
3. **风险可控**：开发成本低，即使失败也损失不大
4. **渐进式**：未来可以升级到 Temporal 或扩展功能

**决策树**：
```
用户需求是什么？
├─ 简单任务串联（顺序执行）
│   └─ 方案C ✅
├─ 复杂工作流（并行、循环、长时间运行）
│   ├─ 能接受后台进程？
│   │   ├─ 是 → 方案A（Temporal）
│   │   └─ 否 → 方案B（自建，但成本高）
│   └─ 不确定 → 先用方案C验证需求
└─ 不确定
    └─ 方案C（快速验证）✅
```

## 下一步行动

1. **与用户确认需求**：
   - Lumos 的工作流主要用于什么场景？
   - 是否需要复杂的并行和循环？
   - 是否需要长时间运行（> 1小时）？
   - 用户能否接受后台进程？

2. **如果选择方案C**：
   - 设计简化版 TaskPlan DSL
   - 实现 TaskExecutor
   - 集成到 Scheduling Layer
   - 提供简单的 UI（列表视图，不需要可视化编辑器）

3. **如果选择方案A**：
   - 调研 Temporal CLI 的嵌入方式
   - 设计进程管理方案
   - 评估资源占用

4. **如果选择方案B**：
   - 详细设计工作流引擎架构
   - 制定测试计划
   - 评估开发资源
