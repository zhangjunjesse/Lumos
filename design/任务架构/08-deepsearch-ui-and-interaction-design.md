# DeepSearch UI 与交互设计文档

## 0. 文档定位

本文补充回答两个落实现前必须说清的问题：

- 用户在正式 `DeepSearch` 页面里到底看到什么、点什么、怎么恢复
- “接管当前活动页 / 新建受管页”“strict / best_effort”“等待登录 / 部分完成”这些设计决策如何落成明确交互

本文不替代以下文档：

- `08-deepsearch-requirements-design.md`
- `08-deepsearch-architecture-design.md`
- `08-deepsearch-phase-1-implementation-design.md`

它的目标是把 `08 DeepSearch` 从架构概念推进到可直接指导产品和前端实现的交互层。

---

## 1. 设计目标

DeepSearch 正式页的交互设计必须同时满足四个目标：

1. 让用户先把登录态、页面上下文和执行严格度准备好，再开始抓取
2. 让用户能清楚知道当前 run 是“等待登录 / 已暂停 / 部分完成 / 已失败”中的哪一种
3. 让用户能直接看到抓取正文、截图、证据和失败原因，而不是只看到一段摘要
4. 让聊天和 Workflow 复用同一批 run/detail/artifact，但不把正式 UI 心智打散

---

## 2. 正式产品入口

Phase 1 正式入口建议固定为：

- 左侧侧边栏
- `扩展`
- `DeepSearch`

建议正式入口落点为：

- `/extensions?tab=deepsearch`

Phase 1 不建议新增单独一级路由如：

- `/deepsearch`

Phase 1 不建议在 `扩展` 体系外再拆新的一级导航或很多子路由。

原因：

- 用户当前心智是“去 DeepSearch 页做登录、发起、看结果”
- 子路由太早拆开会把“配置 / 运行 / 详情”割裂
- Phase 1 先用单页四区结构（顶部任务区 + 左中右三栏）更适合快速验证完整主链

---

## 3. 页面信息架构

Phase 1 建议采用四区布局：

- 顶部：任务发起区
- 左栏：站点连接区
- 中栏：运行记录区
- 右栏：运行详情区

### 3.1 顶部任务发起区

必须包含以下输入项：

- 查询词 / 研究任务描述
- 站点范围
- 研究目标
- 页面来源模式
  - `接管当前活动页`
  - `新建受管页`
- 执行严格度
  - `strict`
  - `best_effort`
- 最大抓取页数
- 最大深度
- 是否保留正文证据
- 是否保留截图

必须包含以下操作：

- `开始 DeepSearch`
- `暂停`
- `恢复`
- `取消`

注意：

- `暂停 / 恢复 / 取消` 只在有活动 run 时显示为可用
- 没有活动 run 时，这三个按钮应明确禁用，而不是消失得毫无提示

### 3.2 左栏站点连接区

建议每个站点卡片至少显示：

- 站点名称
- 当前连接状态
- 最近验证时间
- 最近成功登录时间
- 当前阻塞说明
- `去登录`
- `重新检查`

站点状态文案固定为：

- `未连接`
- `已连接`
- `疑似过期`
- `已失效`
- `检查失败`

### 3.3 中栏运行记录区

每条记录建议显示：

- 查询词或任务标题
- 站点范围
- 页面模式
- 严格度
- 当前状态
- 抓取页数
- 证据数
- 创建时间
- 完成时间

建议支持以下筛选：

- 全部
- 运行中
- 等待登录
- 已暂停
- 部分完成
- 已完成
- 失败

### 3.4 右栏运行详情区

点击中栏任一记录后，右栏应切到该 run 的详情。

建议拆成四个详情分区：

- `摘要`
- `页面结果`
- `证据与产物`
- `失败与阻塞`

如果当前没有选中记录，右栏显示空态：

- `选择一条 DeepSearch 记录查看详情`

## 4. 关键交互流

## 4.1 站点登录检查流

用户路径：

1. 打开 `扩展 -> DeepSearch`
2. 在左栏看到站点卡片
3. 点击 `重新检查`
4. 状态刷新为 `已连接 / 疑似过期 / 已失效 / 检查失败`

交互要求：

- 检查中要显示明确 loading 状态
- 失败时要显示失败原因摘要，不要只显示空白或红点
- 如果检查结果会影响当前运行中的任务，详情区要同步展示“受影响 run”

## 4.2 “接管当前活动页”发起流

这是本轮新增的正式主链，必须有清晰确认交互。

用户路径：

1. 在顶部任务发起区选择 `接管当前活动页`
2. UI 立即显示当前浏览器活动页摘要
   - 页面标题
   - URL
   - 站点域名
3. 用户点击 `开始 DeepSearch`
4. 系统弹出一次轻确认
   - 将接管当前活动页用于本次 DeepSearch
   - 后续证据和失败记录会绑定到本次 run
5. 用户确认后正式开始

交互要求：

- 发起前必须让用户看见“当前将被接管的是哪一页”
- 如果活动页域名与所选站点范围明显不匹配，要给警告，但不必强制阻止
- 一旦开始执行，详情区必须展示该 run 已绑定的活动页标题和 URL

不建议的交互：

- 用户选了“接管当前活动页”，但系统静默改成别的同域 tab
- 用户不知道当前被接管的是哪一页

## 4.3 “新建受管页”发起流

用户路径：

1. 选择 `新建受管页`
2. 设置查询和站点参数
3. 点击 `开始 DeepSearch`
4. 系统在共享浏览器上下文中新建受管页并执行

交互要求：

- 发起后详情区要展示已创建的受管页数量
- 如果后续为多站点或多页面执行，也要让用户能看见当前 run 绑定过哪些页面

## 4.4 等待登录与恢复流

用户路径：

1. run 因关键站点登录失效进入 `等待登录`
2. 中栏状态变为 `等待登录`
3. 右栏详情明确展示
   - 阻塞站点
   - 阻塞原因
   - `去登录`
   - `恢复`
4. 用户点击 `去登录`
5. 在共享浏览器内完成登录
6. 回到 DeepSearch 页点击 `恢复`
7. run 回到 `等待开始` 或 `运行中`

交互要求：

- `等待登录` 不应表现成普通失败 toast
- 要在详情中明确显示“哪个站点阻塞了本次 run”
- `恢复` 按钮只有在登录复检通过后才可点击

## 4.5 暂停、恢复、取消流

用户路径：

1. 运行中的 run 点击 `暂停`
2. 状态变为 `已暂停`
3. 用户可点击 `恢复` 或 `取消`

交互要求：

- `已暂停` 是正式可见状态，不是内部 debug 状态
- 详情区要保留当前 checkpoint 摘要
  - 已完成站点
  - 剩余站点
  - 当前停在什么阶段
- `取消` 后状态变为 `已取消`，且不再显示 `恢复`

## 4.6 `best_effort` 与 `partial` 结果流

用户路径：

1. 用户选择 `best_effort`
2. 某些站点成功，某些站点被登录阻塞或抓取失败
3. 最终 run 状态显示为 `部分完成`
4. 右栏详情展示：
   - 成功站点列表
   - 未完成站点列表
   - `partial` 原因摘要

交互要求：

- `部分完成` 不能只显示黄色 badge
- 详情里必须能直接看出“到底哪里没完成”
- 历史记录里也应保留 `partial` 的抓取页数和证据数，而不是被当成失败丢掉

## 4.7 历史记录重跑流

用户路径：

1. 用户在中栏选中历史记录
2. 点击 `按原参数重跑`
3. 系统用历史参数回填顶部发起区
4. 用户可修改页面模式、严格度、页数深度后再次发起

交互要求：

- 历史参数回填时必须把 `页面模式` 和 `严格度` 一起回填
- 如果原 run 使用的是 `接管当前活动页`，重跑时不要静默复用旧 `pageId`
- 重跑只复用参数，不复用旧页面绑定

## 5. 运行详情设计

## 5.1 摘要分区

建议显示：

- run 状态
- 查询词
- 站点范围
- 页面模式
- 严格度
- 最大页数 / 深度
- 创建时间 / 完成时间
- 抓取页数 / 证据数
- 最终摘要

## 5.2 页面结果分区

建议按页面列表展示：

- 标题
- URL
- 所属站点
- 内容状态
  - `仅列表`
  - `部分正文`
  - `完整正文`
  - `抓取失败`
- 抓取时间
- `查看正文`
- `查看截图`

## 5.3 证据与产物分区

建议展示：

- 正文 artifact
- 截图 artifact
- 结构化结果 JSON
- 关键证据摘录
- 相关页面来源

交互要求：

- 长正文不要直接全量塞在概览区
- 详情里应支持折叠预览 + 单独打开

## 5.4 失败与阻塞分区

建议展示：

- 阻塞站点
- 失败页面
- 失败阶段
- 是否与登录态有关
- 关键错误摘录
- 对应证据

这块必须能回答用户最关心的两个问题：

- 为什么没拿到完整内容
- 现在应该去登录、重跑，还是接受 partial

---

## 6. 状态与按钮可用性矩阵

建议用如下规则：

- `等待开始`
  - 可点：`开始 DeepSearch`
- `运行中`
  - 可点：`暂停` `取消`
- `等待登录`
  - 可点：`去登录` `重新检查` `取消`
- `已暂停`
  - 可点：`恢复` `取消`
- `已完成`
  - 可点：`按原参数重跑`
- `部分完成`
  - 可点：`按原参数重跑`
- `失败`
  - 可点：`按原参数重跑`
- `已取消`
  - 可点：`按原参数重跑`

不要做的事情：

- 让 `恢复` 在 `等待登录` 且未复检通过时提前可点
- 让 `取消` 后仍然显示 `恢复`
- 在 `部分完成` 里不显示任何失败上下文

---

## 7. 聊天与 Workflow 的 UI 关系

DeepSearch 正式详情页仍应是结果主入口。

聊天侧建议只显示：

- 简短摘要
- 当前状态
- `查看 DeepSearch 详情`

Workflow 侧建议只显示：

- 调用摘要
- run 状态
- artifact 引用
- `打开 DeepSearch 详情`

不要在聊天或 Workflow 中复制一套完整详情 UI。

原因：

- 正文、截图、失败原因、登录阻塞本来就属于 DeepSearch 自己的产品心智
- 复制多套详情页只会制造信息不一致

---

## 8. Phase 1 UI 严格验收口径

只有同时满足以下条件，才可说 Phase 1 UI 可验收：

1. 用户能打开正式 `DeepSearch` 页面
2. 用户能看到站点登录态并发起登录
3. 用户能选择 `接管当前活动页 / 新建受管页`
4. 用户能选择 `strict / best_effort`
5. 用户能看到 `等待登录 / 已暂停 / 部分完成 / 已完成 / 失败`
6. 用户能在详情区看到正文、截图、证据或失败原因
7. 用户能从 `等待登录` 恢复同一 run
8. 用户能对历史 run 进行重跑

如果还做不到这些，就不能说 `08` 的正式 UI 已经可验收。

---

## 9. 当前结论

一句话总结：

- **DeepSearch 的正式 UI 必须把“接管当前活动页”“等待登录恢复”“best_effort 下的 partial”“正文与失败证据查看”做成用户直接可见、可点、可理解的交互，而不是只存在于 service 内部状态里。**
