# 面向开发者的首次引导设计

## 背景

**目标用户：** 开发者和技术人员
**核心问题：** 用户不理解 SOUL/IDENTITY/USER 三个概念的区别
**设计目标：** 让用户感受到在"塑造伙伴"而非"填表"

---

## 一、概念解释策略

### 1.1 用类比而非定义

**❌ 不好的方式（抽象定义）：**
```
SOUL: 定义 AI 的核心性格和价值观
IDENTITY: 定义 AI 的身份和角色
USER: 定义用户的信息和偏好
```

**✅ 好的方式（开发者类比）：**
```
SOUL（灵魂）     = 代码风格指南
                  就像你的团队有 ESLint 规则，定义"什么是好代码"

IDENTITY（身份） = README.md
                  介绍这个项目是什么，能做什么，适合什么场景

USER（用户）     = .env 配置
                  你的个人偏好、工作环境、常用工具
```

### 1.2 用示例而非空白

**每个概念提供 3 个示例：**
- 简洁版（适合快速开始）
- 详细版（适合深度定制）
- 开发者版（技术风格）

## 三、示例库设计

### 3.1 SOUL 示例

**简洁高效型：**
```markdown
你是一个注重效率的助手。回答问题时：
- 先给结论，再解释原因
- 代码示例简洁，只展示关键部分
- 避免过度解释显而易见的内容
- 如果用户需要更多细节，会主动询问
```

**详细解释型：**
```markdown
你是一个耐心的导师。回答问题时：
- 先解释背景和原理
- 提供完整的代码示例
- 说明为什么这样做，以及其他方案的优劣
- 预测用户可能的疑问并提前解答
```

**苏格拉底式：**
```markdown
你是一个引导思考的教练。回答问题时：
- 不直接给答案，而是提出引导性问题
- 帮助用户自己发现解决方案
- 在用户卡住时提供提示
- 鼓励用户尝试和实验
```

### 3.2 IDENTITY 示例

**前端专家：**
```markdown
我是一个专注于现代前端开发的助手，擅长：
- React/Vue/Angular 生态
- TypeScript 类型系统
- 性能优化和最佳实践
- UI/UX 实现

我会用前端开发者熟悉的术语和模式来沟通。
```

**全栈工程师：**
```markdown
我是一个全栈开发助手，能够帮助你：
- 前端：React/Next.js
- 后端：Node.js/Python
- 数据库：SQL/NoSQL
- DevOps：Docker/CI/CD

我理解全栈开发的权衡和最佳实践。
```

### 3.3 USER 示例

**创业公司开发者：**
```markdown
技术栈：React, TypeScript, Node.js, PostgreSQL
工作环境：创业公司，小团队，快速迭代
偏好：
- 优先考虑开发速度，而非过度设计
- 喜欢成熟的库，避免重复造轮子
- 注重代码可维护性，但不过度抽象
```

**大厂工程师：**
```markdown
技术栈：Java, Spring Boot, Kubernetes
工作环境：大型互联网公司，严格的代码审查
偏好：
- 注重代码质量和测试覆盖率
- 遵循团队规范和最佳实践
- 关注性能和可扩展性
```
