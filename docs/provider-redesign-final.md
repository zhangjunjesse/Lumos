# Claude 配置页面重新设计 - 最终产品文档

## 1. 设计目标

基于任务 #1 的问题分析，本次重新设计的核心目标：

1. **简化认知**：去掉 12 个预设选项的"添加提供商"区域
2. **提升透明度**：清晰显示当前配置和连接状态
3. **优化体验**：单列布局，Switch 切换自定义 API
4. **代码重构**：拆分 875 行的超大文件，符合 300 行规范

---

## 2. UX 设计方案

### 2.1 页面重新定位

**新命名**：Claude 配置（Claude Configuration）

**理由**：
- "服务商"过于宽泛，用户不理解
- "Claude 配置"直接说明功能：配置 Claude API
- 与侧边栏导航项保持一致的命名风格

### 2.2 信息架构

**去掉"添加提供商"区域**，改为：

1. **当前配置卡片**（顶部）
   - 显示正在使用的 API 配置
   - 显示连接状态（正常/异常）
   - 显示当前模型

2. **API 密钥管理卡片**（主体）
   - Switch 开关：使用自定义 API Key
   - 折叠表单：API Key + Base URL（可选）
   - 操作按钮：测试连接 + 保存配置

### 2.3 交互流程

**简化配置流程**：
1. 默认使用 Claude Code 内置配置
2. 打开 Switch → 展开表单
3. 输入 API Key → 点击"测试连接"
4. 测试成功 → 点击"保存配置"
5. 关闭 Switch → 恢复内置配置

**核心改变**：
- 不再有"添加提供商"的概念
- 只有"使用自定义 API Key"的开关
- 一次只能配置一个 API（简化选择）

## 3. UI 视觉设计

### 3.1 视觉风格

- **简洁专业**：去除冗余元素，突出核心配置信息
- **层次清晰**：通过视觉权重区分主次信息
- **状态明确**：用颜色和图标清晰表达配置状态
- **响应友好**：适配不同屏幕尺寸

### 3.2 色彩系统

**状态色**：
- Success: `oklch(0.6 0.118 184.704)` - 绿色，表示已配置/连接成功
- Warning: `oklch(0.828 0.189 84.429)` - 橙色，表示需要注意
- Destructive: `oklch(0.577 0.245 27.325)` - 红色，表示错误

### 3.3 组件选择

| 功能 | 组件 | 用途 |
|------|------|------|
| 配置卡片 | `Card` | 包裹当前配置和 API 密钥表单 |
| 状态指示 | `Badge` | 显示"已配置"/"未配置"状态 |
| 输入框 | `Input` | API Key 输入（type="password"） |
| 按钮 | `Button` | 主要操作（保存/测试连接） |
| 开关 | `Switch` | 切换"使用自定义 API Key" |
| 折叠面板 | `Collapsible` | 展开/收起 API 密钥表单 |
| 提示信息 | `Alert` | 显示错误/成功消息 |

## 8. 附录

### 8.1 当前问题回顾

参考任务 #1 的详细分析：
- 命名与功能不匹配（"服务商"过于宽泛）
- 添加功能设计混乱（12 个预设选项造成选择困难）
- 信息架构问题（缺少使用状态、配额等关键信息）
- 交互体验问题（流程割裂、逻辑复杂）
- 技术债务（875 行超大文件）

### 8.4 核心设计决策

**为什么去掉多 Provider 管理？**
- 用户实际只需要一个 API 配置
- 多选项造成认知负担和选择困难
- 简化为 Switch 开关更符合用户心智模型

**为什么用 Switch 而不是按钮？**
- Switch 清晰表达"开/关"状态
- 符合"使用自定义配置"的语义
- 减少点击步骤，提升效率

**为什么保留 Base URL？**
- 支持第三方兼容 API（如 OpenRouter）
- 支持企业内网部署
- 折叠在高级选项中，不干扰普通用户
